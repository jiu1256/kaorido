# CLAUDE.md

フレグランスガイドサイト「Kaorido」。サイト本体（Vite + React 19 + Supabase）と商品データ入力の自動化スクリプトを1つのリポジトリで管理する。

## リポジトリ構成

| パス | 内容 |
| --- | --- |
| `src/` | サイト本体・管理画面（`src/Admin.jsx`） |
| `scripts/` | 商品データ入力の自動化（`import-product.mjs`）と検証用（`verify-product.mjs`） |
| `prompts/` | 商品情報抽出プロンプトのテンプレート |
| `docs/` | DBスキーマ等のドキュメント |
| `sillage/` | プロジェクト初期の Vite スキャフォールドの残骸（未使用） |

環境変数は リポジトリ直下の `.env`（git管理外）。キー名は `.env.example` を参照。フロントは `VITE_` プレフィックス付き（anonキー）、スクリプトは `VITE_` なし（service roleキー）で、**スクリプト側に VITE_ プレフィックスは禁止**。

## DB構成

詳細なテーブル定義と抽出JSON→カラム対応は `docs/schema.md` を参照（**スキーマ変更時は必ず docs/schema.md も更新する**）。

| テーブル | 役割 |
| --- | --- |
| `brands` | ブランドマスタ |
| `products` | 商品本体 |
| `tags` | タグマスタ |
| `product_tags` | products ↔ tags の中間テーブル |
| `effects` | 効能マスタ |
| `product_effects` | products ↔ effects の中間テーブル |
| `fragrance_notes` | 香りのノート（トップ／ミドル／ラスト等） |
| `buy_links` | 購入リンク |

### 重要な仕様
- **variants は jsonb** … `products` 内にサイズ違いを格納する。
- **セット商品** … `type = ギフトキット`。ノート（`fragrance_notes`）は持たない。
- **タグ・効能** … 新規作成せず、既存マスタ（`tags` / `effects`）から選択する方針。
- **fragrance_notes の note_type は `top` / `mid` / `base`**（`middle` ではない）。display_order は 0 始まり。

## 商品データ入力の自動化

- 実行（リポジトリ直下から）: `node scripts/import-product.mjs --brand <ブランド名> [--save] [--skip-notes] [--headful] <URL...>`（--save なしは dry-run）
- **--skip-notes** … 抽出ノートを破棄するフラグ。ノートを人力補完する運用のブランド（SHIRO）では必ず付ける。claude-haiku-4-5 は説明文から香料名を拾ってノートに推測分類する癖があり、プロンプトでの矯正が不安定なため（負例・自己チェックを入れても再発）
- **抽出ノートが空のときは fragrance_notes に触れない**（一括貼り付けで人力補完したノートを再取り込みで消さないためのガード）
- **同名の登録済みが複数件ある場合は保存中止**（どれを更新すべきか不定のため。候補idを表示）
- **登録済み商品への --save は注意**: update は desc/tags/effects/variants を抽出結果で**置き換える**。管理画面で人力修正した公開済み商品に実行するとキュレーション内容（variants内のimages配列等も）が失われる。現状は**新規登録用途**として使うこと
- 抽出プロンプトのテンプレート: `prompts/extract-product.md`（`{{既存type一覧}}` 等のプレースホルダにマスタを差し込んで使用）
- 抽出モデル: claude-haiku-4-5。JSONパース失敗時は1回だけリトライ
- ページ取得は Playwright（ヘッドレスChromium）でレンダリング後のHTMLを使用。JS遅延描画のバリアント・価格を拾うため。起動失敗時は素のfetchにフォールバック
- **brand は抽出に含めない** … スクリプト実行時に引数で指定する。一致するブランドが無ければエラー終了（新ブランドは管理画面で先に登録）
- **buy_links は抽出に含めない** … 後続フェーズで楽天/Yahoo APIから取得予定。ただし抽出出力の `official_url`（処理したページのURL）から「公式サイト」行を作る
- price は税込・整数（円）。税抜表示しかない場合は warnings に記載して税抜のまま。価格がページに無い場合は null のまま保存
- 代表値は**最大容量の variant**（products.price/volume と buy_links.current_price）
- tags はマスタ照合後 7 件超なら先頭 7 件に切り詰めて警告表示
- upsert の name 比較は正規化（前後空白除去・連続空白を1つに・全角英数を半角に）して行う。完全一致が無く**類似名（正規化後の部分一致）がある場合は保存せず候補を表示**して人間に判断させる
- volume は小文字 ml に正規化。セット商品の特殊表記はページ表記を優先
- 新規保存時は `is_published=false`（管理画面で確認してから公開）
- **既知の制約（SHIRO）**: item ページのノート三段チャートは KARTE Blocks 注入の**画像**で、HTML/ヘッドレスでは取得不可（bot/セグメント判定とみられる）。**ノートは管理画面の「一括貼り付け」で人力補完**する運用。他ブランドはノートのテキスト掲載が普通なので自動抽出対象。`--headful` は検証用フラグ（恒久パイプラインにはしない）
- **将来用メモ**: 商品ページには香りの持続時間やレビュー情報もあるが、現状のスキーマに保存先がないため抽出対象に含めていない。必要になったらカラム追加とテンプレート拡張をセットで行う

## 管理画面（src/Admin.jsx）

- 商品編集のフレグランスノート欄に**一括貼り付け機能**あり（TOP/MIDDLE/LAST/BASE/トップ/ミドル/ラスト/ベースのラベルを検出して各欄に振り分け。LAST/ラストは base 扱い。区切りは読点・カンマ・中黒・スラッシュ・スペース。フォーム反映のみで保存は既存の更新ボタン）

## 過去のバグ（再発防止）

- **useEffect の import 漏れ** … React コンポーネントで `useEffect` を使う際は import を確認する。
- **スキーマとフロントの不一致による PATCH 400** … DBスキーマとフロント側の送信フィールドがずれると PATCH が 400 を返す。変更時は両者の整合を確認する。
- **note_type の表記不一致** … import スクリプトが `middle` で保存しかけた（正しくは `mid`）。新規コードがDBに書く際は `docs/schema.md` の規約を必ず参照する。
